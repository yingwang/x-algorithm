# 7. 关键设计决策

这一章不再按文件或者阶段做组织，而是把整套系统在设计层面所做的几处关键决策抽出来逐一审视。每一项决策都附带做法、原因与代价三方面的说明，以便读者在自己设计类似系统时能够清晰地理解其中的工程取舍。

## 第一项 · 零手工特征

**做法：** 让 Grok Transformer 直接从用户的动作序列里自学相关性，把人工特征工程的整条链路彻底砍掉。

**原因：** 传统推荐系统里的人工特征工程存在多项长期痛点，这一选择正是为了从根本上规避这些问题。

| 传统做法的代价 | 本仓库的回避方式 |
|---|---|
| 需要维护特征仓库，线上与线下两套实现 | 不再存在，全部交给模型完成 |
| 训练阶段与服务阶段的特征一致性问题 | 输入只剩下"动作序列加上 ID 哈希"，几乎没有分歧的空间 |
| 新增一项特征需要走完整的发布流程 | 新增一种动作类型只需要扩大 `num_actions` 然后重新训练 |
| 特征工程师与算法工程师之间存在隔阂 | 同一个团队对模型直接负责 |

**代价：** 单个特征的"可解释性"会变差，模型呈现为一种相对黑盒的形态。不过仓库通过"多动作输出"这一设计部分地补偿了这一点。读者至少仍然可以看到模型预测出来的"喜欢概率、转推概率、讨厌概率"分别是多少，归因仍然在一定程度上可行。

## 第二项 · 候选隔离的注意力掩码

**做法：** 在排序 Transformer 内部，让候选之间彼此不可见。注意力掩码只在对角线位置加上一个 1，让每一条候选只能看到 user 段、history 段以及自己。

**收益：** 这一设计同时带来了四项明显的工程价值。一是得分稳定，候选 A 的分数不会受同批中有没有候选 B、C、D 的影响。二是可缓存，`(user, post) → score` 这样的键值对可以直接进缓存，不会出现"批组合"带来的键空间爆炸。三是可并行，候选之间没有依赖，可以直接做流水线化。四是可解释，分数发生变化时归因空间只剩下"用户上下文"和"候选自身"两类原因。

**代价：** 失去了 listwise 信号，也就是"候选间相对优劣"这一层信息。不过 X 这一产品所追求的是"绝对相关性"而非"列表序优劣"，因此这一取舍在产品层面是相当划算的。

更详细的讨论参见[排序 Transformer 与候选隔离](phoenix-ranking.md)一节。

## 第三项 · Hash-Based Embedding

**做法：** 用户、物品、作者等所有 ID 类特征都通过多哈希查表实现 embedding，而不再采用"每一个 ID 单独一行 embedding"的传统做法。

**收益：** 这一设计的好处可以从四个角度来概括。

第一是不存在 OOV。新出现的用户或者帖子天然就具备 embedding：它们会通过哈希函数被映射到某些已经训练好的桶上。

第二是显存占用固定。词表的逻辑大小可以无限大，但是 embedding 表本身的物理大小是固定的，默认 1M 行 × 128 维 × 三张表合起来大约 1.5 GB。

第三是不需要维护"ID 到行号"的映射服务。哈希函数本身就承担了映射的角色，工程上少了一个独立的服务依赖。

第四是冲突概率指数级下降。每一个 ID 使用两个独立的哈希函数加上线性投影，单一冲突的概率因此从 `1/V` 降到了大致 `1/V²` 的量级。

**代价：** 哈希冲突理论上不可避免。但实际中，在两个哈希函数加上每张表 1M 桶的配置下，"两个完全不同的 ID 同时在两张表上都落到同一个桶"的概率小到完全可以忽略。

更详细的讨论参见 [Mini 模型规格](phoenix-mini-specs.md)以及 `phoenix/run_pipeline.py` 中的 `_hash_ids` 函数。

## 第四项 · 多动作预测

**做法：** 排序模型一次性输出 19 个离散动作的概率以及 8 个连续值，下游再通过加权聚合得到单一分数。

**收益：** 这种设计带来的好处主要体现在四方面。

第一是更真实的偏好建模。用户喜欢一条帖子的原因可能是想要点赞、可能是想要转推、也可能是觉得视频好看；这些原因在层次上各不相同，单一"相关性"分数会把它们全部抹平。

第二是负权显式表达"讨厌"。`not_interested`、`block`、`mute`、`report` 这一类负向动作直接通过负权拉低分数，模型不再需要"装作只关心喜欢"。

第三是业务目标可调节。如果产品希望多曝光转推，只需要提高 `RETWEET_WEIGHT` 一行配置即可；如果希望压制短视频，降低 `VQV_WEIGHT` 同样可以做到。

第四是 A/B 实验的解释力强。在做实验对比时可以直接观察哪一个动作的概率发生了变化，从而判断模型在哪一个维度上确实发生了变动。

更详细的讨论参见[多动作预测与加权打分](phoenix-multiaction.md)一节。

## 第五项 · 可组合的 Pipeline 框架

**做法：** `xai_candidate_pipeline` 这一 crate 把流水线抽象成 Source、Hydrator、Filter、Scorer、Selector、SideEffect 等共计七类 trait，业务侧负责实现 trait，框架负责并发调度。

**收益：** 这种框架化的设计同时实现了五项工程目标。

第一，新增候选源时不需要改动任何调度代码，只需要实现 `Source` trait，然后把新组件加进 `vec![]` 装配清单即可。

第二，单一组件失败不会传染。框架对所有 `Err` 静默吞掉，主链路继续运行。

第三，监控开箱即用。阶段级的延迟、计数、过滤率等指标都会被框架自动记录。

第四，A/B 与 feature switch 走统一接口。每一个组件都提供一个 `enable(&query)` 钩子，由 `Decider` 与 `Params` 决定是否启用。

第五，副作用不阻塞主路径。所有副作用都通过 `tokio::spawn` 后台执行。

**代价：** trait 抽象带来一定的学习曲线，新加入的组件必须按 trait 形状实现，初次接触可能需要花时间适应。

更详细的讨论参见 [Candidate Pipeline](components-pipeline.md)一节。

## 第六项 · 服务层全部用 Rust 重写

**做法：** 编排层、候选库、流水线框架全部用 Rust 实现，模型层则留在 Python 生态。

**原因：** Rust 在这一场景下的优势可以这样概括。

第一是内存安全加上没有 GC 停顿，对 Home Mixer 这一类对 P99 延迟要求极为严格的服务而言至关重要。

第二是零成本异步。Tokio 加 tonic 的吞吐量、延迟、资源占比都远优于 JVM 系。

第三是 DashMap 这一类无锁并发数据结构让 Thunder 的内存索引可以在多核上获得近线性的扩展能力。

第四是数据结构的精细控制力。TinyPost、VecDeque、Bloom Filter 这一类低层结构在 Rust 里可以非常精确地控制内存布局，对内存占用与缓存友好程度都有显著影响。

模型部分之所以保留 Python 也有两条充分的理由。一是 JAX 与 Haiku 训练栈已经非常成熟，Python 在机器学习领域是事实上的标准；二是模型部署在独立的服务里运行，通过 `PhoenixPredictionClient` 远程调用，Rust 这一侧根本不需要做 Python 与 Rust 之间的 interop。

## 第七项 · 多源召回加上单一 Transformer 排序

**做法：** 召回侧有六路（Thunder、Phoenix 的多种召回路径、TweetMixer、缓存源等），但是排序只用一个模型。

**原因：** 多源召回追求的是覆盖度。不同源各有优势，相互之间形成互补；任何单一源出现故障时其它源仍能兜底。单一排序模型追求的是一致性，让所有候选都被同一个模型按同一标准打分，避免出现"in-network 用模型 A、out-of-network 用模型 B 导致结果不可比"的情况。

这是经典推荐系统中"召回与精排"的标准分工，本仓库在这一点上做到了相对纯粹的实现。

## 第八项 · 前过滤与后过滤分两段

**做法：** 14 个 filter 放在打分之前，3 个 VF 与 Conversation 相关的 filter 放在选择之后。

**原因：** 这一分段方式的核心是平衡"打分算力"与"VF 服务延迟"两方面的工程代价。前过滤把绝大部分候选剔除掉以节省 Phoenix 推理算力；后过滤只在 Top-K 这几十条候选上做，避免 VF 服务的高延迟拖累整次请求。

更详细的讨论参见[过滤规则](filters.md)一节的末段。

## 第九项 · 副作用全部异步

**做法：** Kafka 写入、Redis 缓存、实验对照、已展示历史更新等全部通过 `tokio::spawn` 异步执行。

**原因：** 让用户感知到的"出 Feed 速度"与"副作用是否完成"完全解耦。即便 Kafka 在某一刻彻底宕机，用户照样能够看到 Feed，只不过后台的离线分析会缺少这一段时间的数据。这种容错策略对于一个高可用服务来说非常重要。

**代价：** 一旦通过 `tokio::spawn` 把任务发出去之后，发生失败时没人会再去接住它（除了一些 metrics 上报）。不过这一类 side effect 本身就是"尽力而为"性质的，并不影响线上的功能正确性。

## 第十项 · 把"广告插入"放到外层 Pipeline

**做法：** 核心的 `PhoenixCandidatePipeline` 只关心有机帖，外层的 `ForYouCandidatePipeline` 再把核心结果与广告、Who-To-Follow、Prompts 等用 `BlenderSelector` 混排。

**原因：** 这种拆分方式同时实现了三项目标。

第一是关注点分离。广告与商业化策略可以独立修改，不需要触碰核心排序逻辑。

第二是品牌安全可控。`SafeGapBlender` 在敏感内容附近不会插广告，具体规则定义在 `home-mixer/ads/` 目录下。

第三是复用性。`ScoredPostsService` 可以被其它面板（例如 Search Home、Profile 推荐）复用，并且不携带广告。

---

## 这些决策合起来想达成什么

如果用一句话概括所有这些决策背后的统一思路，可以这样表述：让"相关性"成为一个可观测、可控、可演进的子系统，把所有非相关性的东西（广告、可见性、商业化）放到外面层叠组合。

这与"端到端一个大模型搞定一切"是两条截然不同的路径。本仓库选择的是"模型负责相关性，框架负责一切其它"。代价是框架本身相对复杂，需要花时间理解；收益是任何一个环节都能独立修改、独立测试、独立监控，长期演进的成本因此大大降低。
