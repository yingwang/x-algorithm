# 1. 项目背景与整体定位

X（前 Twitter）的"For You"信息流需要回答这样一个问题：每当用户下拉刷新一次，系统应当从全平台数以亿计的帖子里选出几十条最值得展示给他的内容，整个过程的端到端延迟必须控制在几百毫秒之内。这是一个量级非常大、对延迟非常敏感、对相关性又有极高要求的工程问题。

为了让这个问题在工程上可解，整个推荐系统把它拆成了两个相对独立的子问题。第一个子问题叫做召回，英文里称作 retrieval。召回阶段的任务，是从全量语料里按照相关性的粗略估计快速截取出几百到几千条候选。这一阶段必须非常便宜，因为输入候选池实在太大，不可能对每一条都进行精细的计算。为了让代价控制在一个可以接受的水平，整套召回逻辑采用的是双塔结构加上近似最近邻搜索的方案。简单来说，模型把用户的兴趣压缩成一个固定维度的向量，再把每一条帖子的内容也压缩成同样维度的向量，两者通过点积来度量相关性。剩下的事情就交给近似最近邻索引去解决，从亿级别的库里取出最接近的几千条只需要毫秒级别的时间。

第二个子问题叫做精排，英文里称作 ranking。精排阶段拿到的是召回阶段截出来的那几千条候选，需要在更精细的颗粒度上对它们重新打分排序。这一阶段的模型可以重得多，使用的是一种 Grok 风格的 Transformer。它不再单纯输出一个标量分数，而是同时预测用户对每一条候选可能采取的多种动作（点击、点赞、转发、停留时间、不喜欢等等）的概率，然后按一组事先定义好的权重把这些概率组合成一个最终的排序分。

## 本仓库的几个特点

把这套设计与传统推荐系统作对比，会发现几处颇为不同的取舍。

第一个特点是，几乎完全去掉了人工特征工程这一层。传统推荐系统里，工程师会花大量时间设计、维护和上线各种各样的人工特征，例如"用户最近七天对某个话题的点击率"或者"作者过去三十天的平均点赞率"。这一类特征往往要靠庞大的特征仓库来管理，离线训练时与在线服务时的取值还要严格对齐，整条链路又长又脆。本仓库的做法是把这一切都交给 Phoenix Transformer 自己去学：模型直接从用户最近的动作序列里推断他对什么内容感兴趣，不再依赖任何手工设计的特征。

第二个特点叫做候选隔离，英文里写作 candidate isolation。在传统的排序模型里，同一批送进来打分的候选之间往往会发生互相影响：换一组候选送进去，原本那条候选的分数也会跟着变化。这种特性会带来两个麻烦，一是分数不稳定，二是缓存几乎无从谈起。本仓库的精排模型通过精心设计的注意力掩码强行把这一影响阻断：每一条候选在注意力计算中只能看到用户上下文，看不到同批的其它候选。这样一来，单条候选的得分就只取决于用户当下的状态，可以独立缓存、跨批稳定。

第三个特点体现在工程实现的语言选择上。承担在线编排任务的 Home Mixer 服务用 Rust 重写，强调的是性能与内存安全。模型本身则继续留在 Python 生态，使用 JAX 与 Haiku 框架。两层之间通过 gRPC 通信。整个候选流水线被进一步抽象成了一套 trait 框架，每一个阶段都可以独立扩展，新的候选源、过滤规则、打分器都可以挂载进来而不破坏现有结构。

第四个特点是仓库内部自带了一份可以直接跑通的最小化样例。仓库里包含一个预训练好的 Mini Phoenix 模型，以及一份体育领域的小语料库。借助这两件资产，读者在本地一台机器上就能完整跑通"召回再到排序"的整条流程，亲手观察每一阶段的输入与输出。这一点对于初次接触工业级推荐系统的读者来说非常重要，毕竟仅靠文字描述是很难真正理解这套系统是如何运作的。

## 为什么这样设计

每一种取舍背后都对应着一种传统做法的痛点。下面这张表把两者并列起来，可以更清楚地看出本仓库选择的方向。

| 传统做法的痛点 | 本仓库的应对 |
|---|---|
| 手工特征链路冗长，需要维护特征仓库，需要保证训练与服务的一致性，迭代成本很高 | 把所有特征生成的工作都交给 Transformer，砍掉整条人工特征工程链路 |
| 同批候选之间互相影响，排序分数不稳定，几乎无法缓存 | 利用注意力掩码强制实现"候选隔离"，让每一条候选的分数只取决于用户上下文 |
| 每一个内容 ID 都要单独维护一个 embedding，词表很快爆炸，新内容还会面临冷启动问题 | 采用多哈希 embedding 的方案，词表大小固定，从根本上消除了未登录词的问题 |
| 单一的相关性分数难以表达"用户也讨厌什么" | 同时预测多种动作的概率，对负向动作显式赋予负权重，让"不喜欢"也参与排序 |
