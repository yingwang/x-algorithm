# 1. 项目背景与整体定位

## 这是一个什么样的问题

X（前 Twitter）的"For You"信息流需要回答这样一个核心问题：每当用户下拉刷新一次，系统应当从全平台数以亿计的帖子中选出几十条最值得展示给他的内容，并且整个过程的端到端延迟必须控制在几百毫秒之内。这是一个量级非常大、对延迟非常敏感、对相关性又有极高要求的工程问题。

把这一问题放到更宽的视野里看，它属于"推荐系统"这一类经典任务。推荐系统的标准设定是：给定一位用户与一个内容库，系统需要给出一个有序的推荐列表，使得用户对靠前的内容产生互动的概率尽可能高。视频网站、电商平台、新闻应用都在解决形态类似的问题，只不过 X 这一具体场景具备两点鲜明的特征：第一，候选语料的规模极其庞大，全平台帖子数轻易就达到亿级别；第二，内容时效性极强，几个小时之前热门的帖子到当下可能就已经不再值得展示。

为了让这个问题在工程上变得可解，整个推荐系统采用了一种业界几乎成为共识的拆分方式：把它拆成召回与排序两个相对独立的子问题。这种拆分的好处在于让每一个阶段都可以采用与自身特点相匹配的算法与数据结构，而不必让单一模型去同时承担"快、准、全"这三件矛盾的事情。

## 召回阶段（Retrieval）

召回阶段所做的事情，可以用一句话概括为"从全量语料里按相关性的粗略估计快速截取出几百到几千条候选"。它的输入是当前用户的画像与近期行为，输出是一份按相关性粗排过的候选 ID 列表，规模通常落在几百到几千条之间。

这一阶段的核心约束是"必须非常便宜"。原因不难理解：输入候选池的规模实在太大，亿级别的语料如果对每一条都做一次精细打分，无论 GPU 还是 CPU 都吃不消。因此召回阶段所选用的模型必须满足两条标准：第一，每一条候选的打分代价要尽量低；第二，整个候选库的表示要可以预先离线计算并存入索引，使得线上推理时只需要"查一次索引"即可拿到 Top-K 结果。

为了同时满足这两条标准，本仓库采用的是"双塔加上近似最近邻搜索"这套组合方案。所谓双塔，是一种模型架构：把用户与候选分别交给两个独立的子网络去编码，最终得到两个固定维度的向量，再通过这两个向量的点积来度量用户与候选之间的相关性。用户那一侧的子网络叫做"用户塔"（User Tower），候选那一侧的子网络叫做"候选塔"（Candidate Tower）。两座塔互相独立，这一独立性正是双塔架构得以高效服务大规模候选库的关键。

近似最近邻搜索（Approximate Nearest Neighbor，简称 ANN）则是与双塔架构搭档使用的检索技术。给定一个查询向量与一个由 N 个向量构成的索引库，ANN 算法可以在远低于 N 的代价之内找出与查询向量最接近的若干向量，所返回的结果与真正的最近邻在很大概率上一致。常见的 ANN 库包括 FAISS（来自 Meta）、ScaNN（来自 Google）等等。借助 ANN，召回阶段可以在亿级语料上做到毫秒级查询。

## 排序阶段（Ranking）

排序阶段所做的事情则与召回完全不同。它的输入是召回阶段截出来的几百到几千条候选，输出则是这一批候选被精细重排之后的有序列表。这一阶段不再追求"快得离谱"，而是追求"算得精细"，因为整个 Feed 是否好用，最终就取决于这一阶段输出的前几条是否真的击中了用户的兴趣。

本仓库的排序阶段使用的是一种"Grok 风格的 Transformer"。这一术语听起来稍显陌生，可以这样理解：Transformer 是当前大语言模型领域最为通用的网络结构，它通过"自注意力"机制让序列中的每一个 token 都可以参考序列里其它 token 的信息；Grok 则是 xAI 公司公开的一族大语言模型；本仓库的排序模型借用了 Grok 的主干结构，但在头部、注意力掩码、损失函数三处做了与推荐场景相关的改造。

排序模型输出的不仅是单一的"相关性分数"，而是同时预测用户对每一条候选可能采取的多种动作的概率，例如点赞、转推、点击、长停留等等。这一组多动作概率随后通过一组预设的权重被组合成最终的排序分。这种"多动作多任务学习"的做法相比传统单一相关性打分有着显著的优势，这一点会在第 5 章里详细展开。

## 本仓库的几个特点

把这套设计与传统的推荐系统作对比，会发现几处颇为不同的取舍。下面把这些特点分别展开来谈。

### 特点一：几乎完全去掉了人工特征工程

传统推荐系统里，算法工程师会花大量时间设计、维护和上线各种各样的人工特征。例如"用户最近七天对某个话题的点击率"、"作者过去三十天的平均点赞率"、"用户与作者之间的好友交集数"等等。这些特征往往需要靠庞大的特征仓库（feature store）来集中管理，离线训练时所看到的特征取值与线上服务时所看到的特征取值还必须严格对齐，否则就会出现"训练服务一致性"问题，让模型上线后表现远不及离线评估。维护这样一整条特征流水线的成本极高。

本仓库的做法是把这一切都交给排序 Transformer 自己去学。模型接收的输入只剩下一段"用户最近的动作序列"，加上每一条候选的若干基础属性（帖子 ID、作者 ID、产品面、帖龄等）。模型的训练目标是预测用户对每一条候选可能产生的多种动作的概率，因此它必须自己从动作序列里推断出"这个用户最近对什么类型的内容感兴趣"。在足够大的训练数据下，Transformer 完全可以学到比手工特征更丰富的内部表示。

### 特点二：候选隔离的注意力掩码

在传统的排序模型里，同一批送进来打分的候选之间常常会发生相互影响。如果模型输入是"用户 + 一批候选"，模型内部就有可能让候选之间的注意力相互影响，从而让某一条候选的分数不仅取决于它自身的特征，还取决于"同批里还有谁"。这种特性在工程上会带来两类麻烦。第一类是分数不稳定：换一批候选送进去，原本那一条候选的分数也会跟着变化。第二类是无法缓存：因为分数依赖于完整的批组合，所以缓存键就要包含批里的全部候选 ID，键空间立刻爆炸。

本仓库的精排模型通过精心设计的"注意力掩码"强行把候选之间的相互影响阻断。注意力掩码（attention mask）是 Transformer 中的一个关键机制，它决定了"每一个 token 在计算自注意力时可以看到序列中的哪些其它 token"。通过把"候选 token 与候选 token 之间的注意力位置全部置零"，本仓库实现了一种叫做"候选隔离"（Candidate Isolation）的效果：每一条候选在注意力计算中只能看到用户与历史 token，看不到同批的其它候选。这样一来，每一条候选的得分就只取决于用户当下的状态加上候选自身的特征，可以独立缓存，分数也跨批稳定。

### 特点三：服务层用 Rust 重写，模型层留在 Python

承担在线编排任务的 Home Mixer 服务用 Rust 重写。Rust 是一种系统编程语言，它的特点是内存安全、没有垃圾回收（GC）造成的延迟抖动、并发模型成本极低。这些特点对于一个需要承担高 QPS 加上严格 P99 延迟约束的在线服务来说几乎是必备的。

模型本身则继续留在 Python 生态，使用 JAX 与 Haiku 这两个框架。JAX 是 Google 推出的数值计算框架，它把 NumPy 风格的接口与自动微分、JIT 编译、加速器（GPU/TPU）支持结合起来；Haiku 则是 DeepMind 在 JAX 之上做的神经网络层抽象库，让模型代码更像传统的 PyTorch 风格。

两层之间通过 gRPC 通信。整个候选流水线被进一步抽象成了一套基于 Rust trait 的框架，每一个阶段都可以独立扩展，新的候选源、过滤规则、打分器都可以挂载进来而不破坏现有结构。这种"语言适合任务"的取舍是大规模工程系统的常见做法。

### 特点四：仓库内部自带一份可以直接跑通的最小化样例

仓库里包含一份预训练好的"Mini Phoenix"模型，以及一份体育领域的小语料库。借助这两件资产，读者在本地一台机器上就能完整跑通"召回到排序"的整条流程，亲手观察每一阶段的输入与输出。

这一点对于初次接触工业级推荐系统的读者来说极为重要。仅仅靠阅读代码或者文字描述很难真正理解一套系统是如何运转的，必须真的把它跑起来、亲手打几个 print 才能形成具体认识。绝大多数公开的推荐系统代码都不附带模型权重，使得它们只能被当作"代码阅读材料"而不能被当作"可运行样例"，这是本仓库相对来说一项很难得的优势。

## 为什么这样设计

每一种取舍背后都对应着一种传统做法的痛点。下面这张表把两者并列起来，可以更清楚地看出本仓库选择的方向。

| 传统做法的痛点 | 本仓库的应对 |
|---|---|
| 手工特征链路冗长，需要维护特征仓库，需要保证训练与服务的一致性，迭代成本很高 | 把所有特征生成的工作都交给 Transformer，砍掉整条人工特征工程链路 |
| 同批候选之间互相影响，排序分数不稳定，几乎无法缓存 | 利用注意力掩码强制实现"候选隔离"，让每一条候选的分数只取决于用户上下文 |
| 每一个内容 ID 都要单独维护一个 embedding，词表很快爆炸，新内容还会面临冷启动问题 | 采用多哈希 embedding 的方案，词表大小固定，从根本上消除了未登录词的问题 |
| 单一的相关性分数难以表达"用户也讨厌什么" | 同时预测多种动作的概率，对负向动作显式赋予负权重，让"不喜欢"也参与排序 |

## 阅读这本书的建议

接下来的几章会逐一展开介绍各个层面的细节。第 2 章先看顶层架构的整体面貌，把整套系统的骨架描清楚。第 3 章深入到五个主要组件的内部，分别讲 Home Mixer、Thunder、Phoenix、Pipeline 框架与 Grox。第 4 章跟随一条真实请求走完完整流水线的每一个阶段，把所有组件串成一条连贯的故事线。第 5 章再回过头来仔细审视 Phoenix 模型本身，深入到双塔、注意力掩码、多动作头等机器学习层面的细节。第 6 章讲过滤规则，第 7 章总结所有关键设计决策，第 8 章给出本地运行 Demo 的具体步骤，第 9 章把本仓库与官方 the-algorithm 仓库做对比，第 10 章是术语对照表，第 11 章是最近一次大版本更新的明细。

对于不同的读者，推荐的阅读顺序略有差异。如果是首次接触工业级推荐系统，建议按章节顺序通读，前面铺好概念再去看具体实现。如果已经有推荐系统经验，希望尽快理解本仓库与官方版本的差异，可以从第 9 章入手，再选读第 5 章了解 Phoenix 模型的具体形态。如果只想跑通 Demo，可以直接跳到第 8 章。
