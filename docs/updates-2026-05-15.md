# 11. 2026-05-15 更新明细

这一次更新把仓库从原本的"代码骨架"提升到了"开箱可跑、含内容理解、带广告变现"的完整版本。本章把每一条更新对应到具体的文件改动，并解释这一条更新在系统层面具有什么样的实际意义。

## 一 · 端到端推理脚本

**新增：** [`phoenix/run_pipeline.py`](https://github.com/yingwang/x-algorithm/blob/main/phoenix/run_pipeline.py)

**替换：** 原先分开的两份脚本 `phoenix/run_retrieval.py` 与 `phoenix/run_ranker.py`，旧版仍然保留在仓库中但不再推荐使用。

**做了什么：** 整合之后只需要一行命令即可完成完整的推理。

```bash
uv run run_pipeline.py --artifacts_dir artifacts/oss-phoenix-artifacts
```

这一行命令的执行顺序与生产链路完全一致：先做召回，再做排序。详细的执行流程参见[本地运行 Phoenix 推理](running.md)一章。

**为什么有意义：** 这一改动让外部读者真的能够把 Demo 跑通，而不是只能阅读代码去猜测模型的实际行为。能够运行的样例与单纯能读到的代码之间存在一道很大的鸿沟，跨过这一鸿沟之后理解的深度会上一个台阶。

## 二 · 预训练 Mini 模型 artifact

**新增：** `phoenix/artifacts/oss-phoenix-artifacts.zip`，体积约 3 GB，通过 Git LFS 分发。

这一压缩包内包含下面这些内容。

- 召回模型的 checkpoint（Transformer 加上 Candidate Tower）
- 排序模型的 checkpoint（Transformer 加上 19 个动作头与 8 个连续头）
- 用户、物品、作者三套多哈希 embedding 表，每一张 1M 行 × 128 维
- 53.7 万条 sports 帖的预计算 candidate embedding 索引（`sports_corpus.npz`）
- 一份示例用户互动序列（`example_sequence.json`）

**为什么有意义：** 长期以来，开源推荐系统几乎不附带模型权重。这一仓库是少数能够开箱进行推理的开源案例。模型权重的存在让 Demo 真正变得"可运行"，而非"可阅读"。

## 三 · Grox 内容理解服务

**新增：** [`grox/`](https://github.com/yingwang/x-algorithm/tree/main/grox)

这一服务包含分类器、嵌入器以及任务调度引擎，覆盖了一系列与内容理解相关的能力。

- 垃圾信息识别（`task_spam_detection.py`）
- PTOS 策略与品类（`task_safety_ptos_policy.py` 与 `task_safety_ptos_category.py`）
- 综合安全筛查（`task_post_safety_screen_deluxe.py`）
- 多模态 embedding（`task_multimodal_post_embedding.py` 加上对应的 sink）
- 摘要再到 embedding 的链路（`task_summarizer_for_post_embedding.py`）
- 基于 Grok 的 UPA 动作打标（`task_grok_upa_action_with_labels.py`）
- ASR、媒体、回复排序、速率限制等其它能力

更详细的讨论参见 [Grox 服务](components-grox.md)一节。

**为什么有意义：** 把内容理解从原本"散落在各处"的状态抽象成独立服务，让 Home Mixer 可以通过下游 store 拿到现成的安全标签、品类、embedding，不再需要自己重新计算。这种解耦让两类系统可以独立演进，团队分工更加清晰。

## 四 · 广告插入与品牌安全

**新增：** [`home-mixer/ads/`](https://github.com/yingwang/x-algorithm/tree/main/home-mixer/ads)

这一目录下有三个关键文件，各自承担不同职责。

- `safe_gap_blender.rs`：处理广告与敏感内容之间的安全间隔，避免广告出现在血腥或争议帖旁边
- `partition_organic_blender.rs`：有机帖与广告的混排策略
- `util.rs`：插入位置工具

外层 `ForYouCandidatePipeline` 通过 `AdsSource` 拉广告，然后由 `BlenderSelector` 调用上面的 blender 决定具体插入位置。这一段流程的详细描述参见 [Home Mixer 编排层](components-home-mixer.md)一节中"两条嵌套 Pipeline"这一小节。

**为什么有意义：** 把广告与品牌安全独立成模块之后，与核心推荐解耦，让广告策略可以单独迭代而不影响有机帖排序。这种关注点分离的做法在大公司里非常常见，但很少能在开源代码里看到完整范例。

## 五 · Query Hydrators 扩充

**新增与增强（位于 `home-mixer/query_hydrators/` 目录下）：**

| Query Hydrator | 拉什么 |
|---|---|
| `FollowedGrokTopicsQueryHydrator` | 用户关注的 Grok 话题 |
| `FollowedStarterPacksQueryHydrator` | 加入的 Starter Pack |
| `InferredGrokTopicsQueryHydrator` | 根据行为推断的兴趣话题 |
| `ImpressionBloomFilterQueryHydrator` | 已展示帖 Bloom Filter |
| `IpQueryHydrator` | IP 到地理位置 |
| `MutualFollowQueryHydrator` | 互关图 |
| `UserDemographicsQueryHydrator` | 年龄、地区、语言 |
| `UserInferredGenderQueryHydrator` | 推断的性别 |
| `ServedHistoryQueryHydrator` | 已展示历史（外层 Pipeline 使用） |
| `PastRequestTimestampsQueryHydrator` | 历史请求时间戳（外层 Pipeline 使用） |

加上原本就已经存在的 Scoring 与 Retrieval Sequence、Blocked、Muted、Followed、Subscribed UserIds、CachedPosts 等 hydrator，整体合计达到 15 个 hydrator 并行执行。

**为什么有意义：** 这些扩充让 Phoenix 模型在构造 user representation 时获得了更全面的"用户画像加上当前上下文"输入信号。模型表现的提升往往不来自更复杂的网络结构，而来自更丰富、更新鲜、更准确的输入信号。

|---|---|
| `GizmoduckCandidateHydrator` | 作者档案 |

| `SubscriptionHydrator` | 付费墙信息 |
| `FilteredTopicsHydrator` | 话题标签 |
| `QuoteHydrator` | 引用帖展开 |
| `HasMediaHydrator` | 是否含有媒体 |
| `LanguageCodeHydrator` | 帖子语言 |
| `VideoDurationCandidateHydrator` | 视频时长，用于 VQV 权重门槛 |
| `AdsBrandSafetyHydrator` 与 `AdsBrandSafetyVfHydrator` | 品牌安全标签（post-selection 阶段） |

**为什么有意义：** 给 scorer 与 filter 提供了更多可用的判断字段，让排序与过滤的决策都能更加精细准确。

## 七 · 新增候选源

**新增（位于 `home-mixer/sources/` 目录下）：**

| Source | 类型 |
|---|---|
| `AdsSource` | 广告（属于外层 Pipeline） |
| `WhoToFollowSource` | "推荐你关注" |
| `PhoenixMOESource` | Phoenix 混合专家召回 |
| `PhoenixTopicsSource` | Phoenix 主题维度召回 |
| `PromptsSource` | 顶部提示卡 |
| `PushToHomeSource` | 推送到 Home 的特殊候选（属于外层 Pipeline） |

同时也刷新了原有的 `ThunderSource`、`PhoenixSource`、`CachedPostsSource`、`TweetMixerSource` 这几路源的实现。

**为什么有意义：** 多路召回带来覆盖度提升；MoE 与 Topics 让 Phoenix 不再只有"双塔召回"这一种模式，从而在召回阶段就具备了更高的多样性。
